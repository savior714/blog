## Cline, Roo Code, Zoo Code, Kilo Code... 다 써봤다.

AI 코딩 에이전트 생태계가 폭발적으로 늘어나는 동안, Cline로 시작해서 Roo Code, Zoo Code, Kilo Code를 넘나들었고, Continue로 IDE 안에서 가볍게 쓰고, Hermes Agent와 PI Agent로 다른 접근법을 시도했고, OMP와 Zed도 경험했습니다.

각각 장단점이 분명했습니다.

Cline는 VS Code 확장으로서의 안정감과 Cline 고유의 워크플로우가 좋았고, Roo Code는 UI가 깔끔하고 설정이 직관적이었습니다. Zoo Code와 Kilo Code는 각각의 차별화된 접근법이 흥미로웠고, Continue는 경량성과 빠른 응답이 매력적이었습니다. Hermes Agent와 PI Agent는 에이전트 아키텍처에 대한 새로운 가능성을 보여줬고, Zed는 속도와 디자인이 일품이었습니다.

그런데 결국 저는 Opencode로 정착했습니다.

## 왜 Opencode인가

### 1. 도구 호출의 신뢰성

에이전트 도구를 쓸 때 가장 중요한 건, 에이전트가 실제로 무언가를 해낼 수 있는지입니다. 프롬프트는 잘 써도, 도구를 호출하다가 실패하면 모든 게 무너집니다.

Cline, Roo Code, Kilo Code 등 여러 도구를 써봤지만, Opencode가 **도구 호출 실패율이 상대적으로 낮았습니다**. 파일 읽기·쓰기, bash 실행, 브라우저 조작 등 주요 도구가 일관되게 작동했고, 실패했을 때도 에러 메시지가 명확해서 재시도 전략을 세우기 쉬웠습니다.

에이전트의 아이디어가 아무리 좋아도, 도구를 못 쓰면 소용없습니다. Opencode는 그 부분을 안정적으로 채워주고 있었습니다.

### 2. 서브에이전트와 오케스트레이션 — 로컬 LLM으로도 큰 컨텍스트를 다루는 법

Opencode의 또 다른 강점은 **subagent 호출과 오케스트레이션**입니다. 복잡한 작업을 맡길 때 서브에이전트를 생성하고, 명확한 계약(handoff contract)을 통해 결과를 받아오는 구조가 잘 설계되어 있습니다.

이게 왜 중요한지 알려면 로컬 LLM의 한계를 이해해야 합니다. 로컬 모델은 클라우드 GPT급 컨텍스트 처리 능력이 없습니다. 긴 컨텍스트를 한 번에 넣으면 품질이 급격히 떨어집니다.

Opencode는 서브에이전트를 통해 작업을 **세분화**합니다. 다른 도구들이 무조건 모든 것을 메인 에이전트가 처리하려 하는 반면, Opencode는 **적절한 타이밍에 서브에이전트로 위임하는 판단**이 더 정확했습니다. 탐색이 필요할 때, 구현이 복잡해질 때, 검증이 필요할 때 — 지능적으로 분기하는 거죠.

로컬 LLM이 감당하기 어려운 큰 컨텍스트도, 이 방식으로 분산하면 상대적으로 쉽게 다룰 수 있습니다. 오케스트레이션이 잘 되어 있어서, 서브에이전트 간 handoff가 매끄럽고 컨텍스트 손실이 최소화됩니다.

실제 경험에서 한 가지 더 발견한 것이 있습니다. 서브에이전트를 병렬로 호출하면 속도는 빠를 수 있지만, 로컬 LLM의 한계상 에러가 훨씬 많았습니다. 컨텍스트가 분산되면서 메인 에이전트의 추적 능력이 떨어지고, handoff 실패나 응답 파싱 오류가 빈번하게 발생했죠. 결국 서브에이전트를 순차적으로 호출하는 게 로컬 LLM 환경에서는 더 안정적이었습니다. 속도는 느리지만, 실패율이 현저히 낮고 결과를 신뢰할 수 있습니다. 병렬 호출이 항상 정답은 아닙니다.

### 3. 오픈소스 + 로컬 실행 — 제 데이터는 내 것

Opencode는 LM Studio나 Ollama와 연결해 로컬 모델로 모든 작업을 수행할 수 있습니다. API 키 없이, 인터넷 없이, 외부로 코드 한 줄 보내지 않죠.

프라이버시와 비용 제로는 이제 선택이 아니라 필수입니다.

## 모든 도구에 매력을 느꼈지만

Cline의 안정감, Roo Code의 UI, Hermes agent의 발전성,Zed의 속도 — 이 각각을 완전히 무시할 수는 없습니다. 하지만 **프로젝트 전체를 아우르는 프레임워크**로서는 Opencode가 저에게 가장 만족스러운 기능을 제공하는 프레임워크였습니다.

그리고 Codex CLI가 2026년 6월 중순부터 로컬 LLM 연동을 부분적으로 지원하기 시작해 일단 써봤지만, Opencode를 포함해 저 위에 기재된 프레임워크들과 비교했을 때 아직 만족도가 떨어졌습니다. 추후 기능 강화 후 다시 써보기를 기대합니다.

## 결론: 정착이란 선택의 결과

저는 Opencode가 가장 완벽한 도구라고 생각하지 않습니다. 하지만 현재로서는 local llm에게는 가장 **신뢰를 주는** 도구라고 생각합니다.

도구 호출의 신뢰성, 서브에이전트 오케스트레이션, 로컬 실행 — 이 세 가지가 조합되어, 작은 시작부터 큰 프로젝트까지 이어지는 선상을 만듭니다.
