# Local LLM 비교 영상 기본 대본 템플릿 (v1)

## 1) 이 템플릿의 목적

이 문서는 "로컬 LLM 모델별 차이"를 같은 형식으로 반복 제작하기 위한 기본 대본 템플릿이다.

- 대상 영상 길이: **10~15분**
- 녹화 정책: **수동 ON/OFF만 사용 (상시 녹화 금지)**
- 오디오 정책: **핵심 구간만 수동 더빙 + 나머지 구간 저작권 안전 BGM**
- 화면 정책: **맥북 화면 기준 LM Studio / VSCode 최대 화면 사용**

---

## 2) 사용 방법 (회차 시작 전)

1. 아래 `변수`를 먼저 채운다.
2. `타임라인 대본`에서 회차에 맞게 문장만 다듬는다.
3. `표준 테스트 묶음` 중 이번 회차에 쓸 테스트를 체크한다.
4. 촬영 후 `회차 기록 시트`를 채워 다음 영상 비교 기준으로 남긴다.

---

## 3) 변수 (회차마다 교체)

```text
[EPISODE_NO]            예: 01
[EPISODE_TITLE]         예: Local LLM 비교 #01 - Qwen 3.6 35B A3B Agent Workflow (feat. MacBook M5 Max 128GB)
[MODEL_A]               예: qwen3-8b
[MODEL_B]               예: llama3.1-8b
[MODEL_C_OPTIONAL]      예: mistral-7b (없으면 삭제)
[QUANT_OR_VARIANT]      예: Q4_K_M / fp16
[HARDWARE]              MacBook M5 Max 128GB
[LMSTUDIO_VERSION]      예: 0.x.x
[VSCODE_SETUP]          예: Roo Code + API Proxy
[TASK_DOMAIN]           예: 코드 리팩토링 / 로그 분석 / 문서 요약
[PROMPT_SET_NAME]       예: dev-assistant-benchmark-v1
[TARGET_DURATION]       예: 12분
[BLOG_URL]              예: (게시 후 기입)
```

---

## 3.1 제목 표기 규칙 (권장)

- 유튜브 제목/썸네일/블로그 제목에 하드웨어를 아래 형식으로 통일:
  - `(feat. MacBook M5 Max 128GB)`
- 예시:
  - `Local LLM 비교 #01 - Qwen 3.6 35B A3B (oQ6) + Roo Code Agent (feat. MacBook M5 Max 128GB)`

## 4) 표준 촬영 타임라인 + 대본 템플릿 (10~15분)

## 4.1 0:00~0:40 오프닝 (핵심 더빙)

- 화면: 타이틀 카드 + 오늘 비교 모델명
- 녹화: `ON`
- 대본 템플릿:
  - "이번 영상에서는 `[MODEL_A]`와 `[MODEL_B]`를 같은 조건에서 비교해보겠습니다."
  - "기준은 속도, 응답 품질, 로그 패턴, 그리고 메모리 점유입니다."
  - "끝까지 보시면 어떤 용도에 어떤 모델이 맞는지 바로 선택할 수 있게 정리해드릴게요."

## 4.2 0:40~2:30 환경/조건 공개 (핵심 더빙)

- 화면: LM Studio 설정 화면, 하드웨어/버전 정보, 프리셋 스냅샷
- 대본 템플릿:
  - "비교 공정성을 위해 하드웨어는 `[HARDWARE]`, LM Studio는 `[LMSTUDIO_VERSION]` 기준입니다."
  - "프롬프트 세트는 `[PROMPT_SET_NAME]`, 프리셋은 동일 조건으로 맞췄습니다."
  - "오늘 도메인은 `[TASK_DOMAIN]`입니다."

## 4.3 2:30~5:30 모델 A 시연 (부분 더빙 + BGM)

- 화면: LM Studio에서 `[MODEL_A]` 로드 -> 프롬프트 실행 -> 토큰/로그
- 녹화 큐:
  - 모델 선택 직전 `ON`
  - 로드 완료 후 결과 확인까지 유지
  - 필요 없는 대기 구간은 편집으로 컷
- 대본 템플릿:
  - "[`MODEL_A`] 먼저 보겠습니다. 지금 로드했고, 같은 프롬프트를 넣겠습니다."
  - "토큰 생성 속도는 대략 [A_TOKENS_PER_SEC], 첫 응답 시작은 [A_TTFT] 정도입니다."
  - "로그에서는 [A_LOG_POINT]가 보입니다."

## 4.4 5:30~8:30 모델 B 시연 (부분 더빙 + BGM)

- 화면: LM Studio에서 `[MODEL_B]` 로드 -> 동일 프롬프트 실행 -> 토큰/로그
- 대본 템플릿:
  - "이제 `[MODEL_B]`를 동일 조건으로 실행해보겠습니다."
  - "속도는 [B_TOKENS_PER_SEC], 첫 응답 시작은 [B_TTFT]로 확인됩니다."
  - "로그 특징은 [B_LOG_POINT]로, `[MODEL_A]`와 차이는 [A_B_DIFF_POINT]입니다."

## 4.5 8:30~10:30 VSCode 작업 연동 (핵심 더빙)

- 화면: VSCode Extensions -> Roo Code -> API Proxy -> 실제 작업
- 대본 템플릿:
  - "VSCode에서는 `[VSCODE_SETUP]` 조합으로 모델을 호출합니다."
  - "같은 작업 요청을 넣고, 작업 중 LM Studio 토큰/로그 변화를 같이 보겠습니다."
  - "실사용 감각 기준으로는 [DEV_EXPERIENCE_NOTE]가 중요하게 체감됩니다."

## 4.6 10:30~12:00 메모리 관찰 (핵심 더빙)

- 화면: macOS 활성 상태 보기 > 메모리 탭
- 대본 템플릿:
  - "메모리 관점에서 `[MODEL_A]`는 [A_MEM_GB], `[MODEL_B]`는 [B_MEM_GB] 수준입니다."
  - "메모리 압력/스왑 기준으로 보면 [MEMORY_INTERPRETATION]이라고 해석할 수 있습니다."

## 4.7 12:00~13:30 결론/추천 (핵심 더빙)

- 화면: 비교 요약 카드 (속도/품질/메모리/추천 용도)
- 대본 템플릿:
  - "속도 우선이면 [SPEED_WINNER], 품질 우선이면 [QUALITY_WINNER]가 더 적합했습니다."
  - "오늘 결론은 [FINAL_RECOMMENDATION]입니다."
  - "다음 편에서는 [NEXT_COMPARISON_PLAN]을 비교해보겠습니다."

## 4.8 13:30~끝 CTA (BGM 중심, 짧은 더빙)

- 화면: 블로그 안내, 다음 편 예고
- 대본 템플릿:
  - "상세 로그 해석과 설정값은 블로그에 정리해두겠습니다."
  - "링크는 설명란에 두고, 다음 비교 회차에서 더 깊게 이어가겠습니다."
- 녹화: 끝나면 `OFF`

---

## 5) 표준 테스트 묶음 (회차별 선택)

## 5.1 공통 입력 세트

- 테스트 1: 짧은 코드 생성/수정
- 테스트 2: 오류 로그 원인 분석
- 테스트 3: 긴 문맥 요약
- 테스트 4: 지시 준수(형식/제약 조건)

## 5.2 공통 측정 항목

- TTFT (첫 토큰까지 시간)
- tokens/sec
- 총 응답 시간
- 응답 정확도/일관성(주관 평가 + 체크리스트)
- 메모리 점유(활성 상태 보기)
- 로그 안정성(에러/재시도/경고)

---

## 6) 샷별 자막 템플릿 (짧은 문구)

- 오프닝: "같은 조건, 다른 모델 - 실사용 기준으로 비교"
- 모델 A: "`[MODEL_A]` 로드/실행 결과"
- 모델 B: "`[MODEL_B]` 로드/실행 결과"
- VSCode 연동: "Roo Code + API Proxy 실사용 흐름"
- 메모리 분석: "활성 상태 보기로 보는 실제 점유"
- 결론: "속도 vs 품질 vs 메모리, 이번 회차 결론"

---

## 7) 편집 템플릿 규칙

- 10~15분 초과 시, 아래 순서로 컷:
  1) 대기/로딩 무의미 구간
  2) 중복 설명 구간
  3) 반복 로그 스크롤 구간
- 핵심 더빙 구간:
  - 오프닝 / 조건 공개 / VSCode 연동 / 메모리 해석 / 결론
- BGM 구간:
  - 모델 실행 대기 / 반복 관찰 / 전환 구간
- BGM 주의:
  - 저작권 안전 소스만 사용
  - 필요한 경우 출처/라이선스 표기를 설명란에 추가

---

## 8) 블로그 연계 템플릿

## 8.1 블로그 목차 (영상 챕터와 동일)

1. 비교 목적과 조건
2. 모델 A 결과
3. 모델 B 결과
4. VSCode 연동 체감
5. 메모리/로그 해석
6. 결론 및 추천 시나리오

## 8.2 본문에 반드시 넣을 것

- 사용한 프롬프트 원문
- 프리셋 값 스냅샷
- 로그 스냅샷 + 해석 코멘트
- 메모리 수치 캡처
- 다음 회차 비교 계획
