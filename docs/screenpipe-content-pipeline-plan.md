# Screenpipe 기반 제작 파이프라인 상세 계획서

## 1) 문서 목적

이 문서는 개발 작업 과정을 영상/문서 콘텐츠로 전환하는 반복 가능한 운영 시스템을 정의한다.
핵심 목표는 아래 2가지다.

- 원하는 시점부터 작업 화면을 녹화해, 불필요 구간을 줄이고 핵심 맥락이 살아 있는 영상으로 만든다.
- 같은 소스(작업 기록/영상)에서 유튜브와 블로그를 연결해, 시청자와 독자가 연속적으로 학습할 수 있게 한다.

---

## 2) 목표 / 범위 / 비범위

### 목표

- **영상 제작 효율화**: 녹화 시작 시점 통제, 컷 편집 규칙화, 게시 전 체크리스트 표준화
- **콘텐츠 일관성**: 영상 제목/챕터/자막/설명과 블로그 구조의 1:1 매핑
- **점진적 자동화**: 현재는 수동 더빙 가능 구조로 운영하고, 이후 TTS 대본 낭독으로 자동화 전환

### 범위 (이번 계획서 기준)

- 녹화 시작 트리거 설계
- 원본 영상 -> 편집본 -> 요약본 -> 자막 -> 오디오 -> 업로드 아티팩트 생성 흐름
- 유튜브 게시 후 블로그 연동 규칙
- 운영 지표 및 개선 루프

### 비범위 (현시점)

- 완전 자동 업로드(100% 무인 배포)
- 고급 모션그래픽/브랜딩 패키지 제작
- 실시간 스트리밍 파이프라인

---

## 3) End-to-End 워크플로우 (상위 단계)

1. **녹화 준비**
  - 작업 목표 1~3개를 미리 적고, 녹화 시작 조건을 선언
2. **녹화 실행**
  - 원하는 시점부터 녹화 시작
  - 작업 중 마커(중요 순간/실수/재시도) 기록
3. **컷 편집**
  - 대기/빌드/반복 탐색 구간 제거
  - 학습 가치가 낮은 시행착오 축약
4. **요약/스크립트**
  - 영상 요약, 챕터, 핵심 포인트, 블로그용 개요 동시 생성
5. **자막 + 오디오**
  - 자막 파일 생성/교정
  - 현재: 필요 시 수동 더빙
  - 향후: TTS 낭독 자동화
6. **유튜브 게시**
  - 제목/썸네일/설명/챕터/태그 검수 후 게시
7. **블로그 연결**
  - 영상 기반 상세 글 발행
  - 영상의 챕터와 블로그 섹션을 연결
8. **회고**
  - 조회/시청 유지율/클릭/독자 체류 데이터를 기록하고 다음 영상에 반영

## 4) 단계별 상세 설계

## 4.1 녹화 시작 설계 (원하는 구간부터 시작)

### 기본 녹화 정책 (확정)

- **기본 상태는 항상 OFF (상시 녹화 금지)**
- **사용자가 녹화 버튼을 누른 시점부터만 기록**
- **사용자가 정지 버튼을 누른 시점에서 즉시 종료**
- 세션 외 시간(대기/잡업무/사적 화면)은 원칙적으로 미기록

### 시작 트리거 규칙

- 다음 중 하나가 만족되면 녹화를 시작한다.
  - 기능 구현 시작 직전
  - 버그 재현이 명확해진 시점
  - 코드 리뷰 반영 시작 시점
- 사전 탐색/환경 정리/무의미 대기 시간은 원칙적으로 미녹화

### 시작 직전 체크 (30초)

- 오늘 영상 한 줄 목적
- 이번 영상의 성공 기준(예: 버그 수정 + 테스트 통과)
- 예상 길이 범위(예: 8~15분 편집본 기준)

### 운영 팁

- 장시간 작업은 세션을 분리(예: Part 1 구현, Part 2 테스트/리팩토링)
- 중요한 맥락 전환 때는 짧은 음성 메모 또는 텍스트 마커 추가

## 4.2 컷 편집 규칙

### 컷 대상

- 빌드 대기/패키지 설치 대기/무입력 정적 화면
- 동일한 오류를 반복 재현하는 장면
- 목표와 무관한 사이드 트랙

### 유지 대상

- 문제 정의가 명확해지는 순간
- 오류 원인을 좁히는 핵심 추론
- 최종 해결 코드와 검증 절차

### 편집 아웃풋 기준

- 1분 내 문제/목표 맥락이 드러나야 함
- 매 챕터 끝에서 "무엇이 해결되었는지"가 명확해야 함
- 결과적으로 블로그 섹션 제목으로 재사용 가능해야 함

## 4.3 요약 및 스크립트 생성

### 생성 산출물

- 영상용:
  - 1문장 훅
  - 3~6개 챕터 제목
  - 챕터별 핵심 포인트 1~3개
- 블로그용:
  - 도입(문제/배경)
  - 본문(접근/실행/검증)
  - 결론(배운 점/다음 액션)

### 요약 품질 기준

- "무엇을 했는가"보다 "왜 그렇게 했는가" 비중 강화
- 코드/명령/결과의 인과관계가 보이도록 작성

## 4.4 자막 + 음성(더빙/TTS) 계획

### 현재 운영 (수동 더빙 중심)

- 자동 생성 자막 초안 -> 수동 교정
- 더빙 필요 파트만 선택해 수동 녹음

### 향후 운영 (TTS 자동화)

- 스크립트 템플릿을 TTS 친화 문장으로 변환
- 용어 발음 사전(프로젝트/라이브러리명) 관리
- 품질 기준 충족 시 수동 더빙 비중 축소

### 품질 체크

- 자막 싱크 오차 허용 범위(예: 0.3~0.5초 이내)
- 고유명사 표기 일관성
- 오디오 레벨 편차 최소화

## 4.5 유튜브 게시 설계

### 게시 전 체크리스트

- 제목: 문제 + 해결 + 결과 구조
- 설명란: 요약, 챕터 타임스탬프, 관련 링크(블로그/레포)
- 챕터: 블로그 목차와 동일 계층 유지
- 고정 댓글: 핵심 정정/추가 자료 링크

### 게시 후 체크리스트

- 자막 자동 생성본과 수동 자막 충돌 여부 확인
- 모바일/데스크톱에서 챕터 동작 확인
- 링크 유효성 점검

## 4.6 블로그 연계 설계

### 기본 연결 원칙

- 영상 한 편당 블로그 글 1개를 기본 단위로 운영
- 영상 챕터 제목 == 블로그 h2/h3 구조를 최대한 일치

### 블로그 글 구조 템플릿

- 문제 배경
- 시도한 접근과 실패 포인트
- 최종 해결 및 근거
- 코드/커맨드/결과 캡처
- 확장 아이디어 (다음 편 연결)

### 상호 링크 전략

- 영상 설명란에 블로그 링크
- 블로그 상단에 영상 임베드 및 챕터 인덱스
- 시리즈 글은 이전/다음 편 네비게이션 추가

## 5) 폴더/파일 운영안 (초안)

```text
content-pipeline/
  sessions/
    YYYY-MM-DD-topic/
      raw/                # 원본 녹화
      edits/              # 컷 편집본
      subtitles/          # srt/vtt
      voice/              # 수동 더빙 또는 TTS 출력
      publish/
        youtube.md        # 제목/설명/챕터/태그
        blog-outline.md   # 블로그 개요
      logs/
        session-log.md    # 작업 로그/회고
```

## 6) 운영 루틴 (권장)

### 주간 루틴 예시

- 월~목: 작업 세션 녹화 + 편집 후보 축적
- 금: 1편 게시 + 블로그 발행 + 지표 기록

### 세션 단위 루틴

1. 녹화 시작 전 30초 목표 선언
2. 작업 중 마커 기록
3. 종료 후 10분 내 요약 초안 작성
4. 게시 전 체크리스트 완료

## 7) 핵심 지표 (초기)

- 제작 효율:
  - 원본 대비 편집본 비율
  - 1편 제작 총 소요 시간
- 품질:
  - 평균 시청 지속 시간
  - 챕터별 이탈 구간
  - 블로그 평균 체류 시간
- 전환:
  - 유튜브 -> 블로그 클릭률
  - 블로그 -> 영상 재생 유입률

## 9) 점진적 자동화 로드맵

### Phase 1 (지금)

- 녹화/편집/게시를 수동 중심으로 안정화
- 요약/자막/설명 템플릿 표준화

### Phase 2

- 요약 -> 챕터 -> 블로그 개요 자동 초안화
- 자막 자동 생성 + 반자동 교정

### Phase 3

- TTS 대본 낭독 파이프라인 도입
- 품질 기준 통과 시 수동 더빙 최소화

### Phase 4

- 게시 메타데이터 자동 생성/검증
- 시리즈 단위 리포트 자동화

## 12) 에피소드 v1 촬영 시나리오 (LM Studio + VSCode + Activity Monitor)

### v1 목표

- 로컬 LLM 작업 흐름을 한 편으로 보여준다:
  - LM Studio에서 모델 선택/프리셋/로드/토큰 생성/로그 분석
  - VSCode에서 Roo Code 설치/기본 설정/API proxy 연결/실제 작업 실행
  - 작업 중 LM Studio 토큰 생성 상태와 macOS 메모리 점유를 함께 설명
  - 최종 편집 길이는 10~15분 내외를 목표로 유지

### 권장 챕터 구조

1. 왜 이 구성을 쓰는가 (로컬 실행, 관찰 가능성, 비용/속도 관점)
2. LM Studio 준비: 모델 선택 -> 프리셋 설정 -> 로드
3. LM Studio 런타임 관찰: 토큰 생성과 로그 읽는 법
4. 모델 로드 후 프리셋 수정 및 별도 저장
5. VSCode 준비: Roo Code 설치와 기본 설정
6. API proxy로 모델 연결 후 실제 작업 시연
7. 작업 중 LM Studio 토큰/로그 동시 관찰
8. macOS 활성 상태 보기에서 메모리 점유 해석
9. 정리: 어떤 상황에서 어떤 프리셋/설정을 쓰면 좋은가

### 샷 리스트 (녹화 버튼 수동 ON/OFF 기준)

- **Shot A (LM Studio 초기 설정)**
  - 시작 시점: 모델 선택 UI를 열기 직전 `녹화 ON`
  - 포함 장면: 모델 선택, 프리셋 값 설정, 로드 버튼 실행
  - 종료 시점: 모델 로드 완료 상태 확인 후 `녹화 OFF`
- **Shot B (토큰/로그 분석)**
  - 시작 시점: 추론 요청 직전 `녹화 ON`
  - 포함 장면: 토큰 생성 흐름, 로그 증가, 로그 항목 해석
  - 종료 시점: 핵심 로그 읽기 설명 완료 후 `녹화 OFF`
- **Shot C (모델 로드 이후 프리셋 변경/저장)**
  - 시작 시점: 프리셋 수정 화면 진입 직전 `녹화 ON`
  - 포함 장면: 값 수정, 별도 프리셋 이름으로 저장
  - 종료 시점: 저장 확인 후 `녹화 OFF`
- **Shot D (VSCode + Roo Code 세팅)**
  - 시작 시점: Extensions 화면 진입 직전 `녹화 ON`
  - 포함 장면: Roo Code 설치, 기본 설정, API proxy 연결
  - 종료 시점: 모델 호출 테스트 성공 후 `녹화 OFF`
- **Shot E (실작업 + 병행 관찰)**
  - 시작 시점: 실제 작업 명령 시작 직전 `녹화 ON`
  - 포함 장면: VSCode 작업 진행, LM Studio 토큰/로그 변화 교차 확인
  - 종료 시점: 작업 결과 확인 후 `녹화 OFF`
- **Shot F (macOS 메모리 설명)**
  - 시작 시점: 활성 상태 보기 메모리 탭 진입 직전 `녹화 ON`
  - 포함 장면: LM Studio/VSCode 점유량 확인, 메모리 압력/스왑 설명
  - 종료 시점: 핵심 수치 요약 후 `녹화 OFF`

### 내레이션 포인트 (핵심만)

- LM Studio:
  - "이 프리셋이 출력 스타일/속도/안정성에 어떤 영향을 주는지"
  - "로그에서 지금 병목이 모델 추론인지 I/O인지"
- VSCode:
  - "API proxy 경유 이유(모델 교체 유연성/환경 분리)"
  - "작업 요청 시 토큰 생성량과 응답 품질의 상관관계"
- Activity Monitor:
  - "메모리 압력, 앱 메모리, 스왑이 체감 성능에 주는 영향"

### v1 편집 가이드

- 맥북 화면 기준으로 LM Studio/VSCode 모두 최대 크기 화면을 사용해 가독성을 우선한다
- 최종 영상 길이는 10~15분 내외로 맞추고, 초과 시 반복 구간을 우선 컷한다
- 장면 전환마다 3~5초 요약 자막 삽입
- Shot 단위로 챕터를 만들어 블로그 섹션과 동일한 제목 사용
- 로그/수치 화면은 줌인 또는 하이라이트로 가독성 확보

### v1 오디오 가이드

- 핵심 설명 구간만 수동 더빙을 넣는다
- 나머지 구간은 저작권 안전 BGM을 낮은 볼륨으로 깐다
- 더빙 구간 진입/이탈 시 BGM 볼륨을 내려 음성 전달력을 우선한다
- 게시 전 BGM 라이선스/출처 표기 필요 여부를 확인한다

### v1 게시 산출물

- 유튜브:
  - 제목 후보 2개, 설명란 1개, 챕터 타임스탬프, 고정 댓글
- 블로그:
  - "설정 -> 실행 -> 관찰 -> 해석" 구조의 상세 글 1개
  - 각 챕터에 스크린샷 1~2장 + 핵심 수치/로그 해석
